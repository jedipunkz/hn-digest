今後、Rust で記述される Linux コードはますます増えていくでしょう。
オランダのユトレヒトで開催された世界最大の Rust 言語カンファレンスである Rust Week カンファレンスで、Linux 安定版カーネルのメンテナである Greg Kroah-Hartman 氏は冒頭、次のように述べました。 「カーネルのセキュリティ リストに 1 ～ 2 か月の長い期間を費やした」後、同氏はその点をさらに推し進めました。「私はさらに大胆な発言をして、『あなたは Linux を救うつもりだ』と言うつもりです。」申し訳ありませんが、すべてはあなたの責任です。」
彼が話していたのは、Dirty Frag、Copy Fail、Fragnesia などの重大な Linux セキュリティ ホールが突然大量に発見され、最新の AI バグ検出プログラムのおかげで明らかになったということでした。
その結果、2005 年以来「これまでカーネル セキュリティのバグをすべて見てきた」Kroah-Hartman 氏は、カーネル チームが現在「1 日あたり 13 件の CVE (一般的な脆弱性とエクスポージャー) か何か、何かクレイジーなこと」を発行していると述べました。彼は、Rust は、C の伝統的なエラー処理やリソース管理の落とし穴から生じる一連のバグを排除する数少ない現実的な方法の 1 つであると考えています。
また: 私がこれまで使用した中で最もユーザーフレンドリーな Linux ディストリビューションの 1 つは、最も安全な Linux ディストリビューションの 1 つでもあります。
Kroah-Hartman 氏は、ポインターをチェックせずに逆参照する 15 年前の Bluetooth のバグや、エラー パスで「ロックを解除するのを忘れた」Xen のバグなど、カーネル内の実際の C のバグに関する落とし穴を説明しました。 「バグの大部分は

「カーネルにあるのは、非常に小さな、マイナーなものです。エラー条件はチェックされず、ロックは忘れられず、未解放のメモリがリークされ、時間の経過とともに脆弱性が増加します。」と彼は説明しました。彼らはカーネルをクラッシュさせます。これが私たちがC言語で生きていることです。これが私たちがそれを好まない理由です。」
Kroah-Hartman 氏は、「Rust の最大の利点」は、レビュー時ではなくビルド時にそれらのミスを発見できることだと主張しました。たとえば、ロックに関して、彼はカーネルにおける Rust のロック抽象化を強調し、「構造体の内部ポインタにアクセスできる唯一の方法は、そのロックを取得し、自動的にロックを解放することです。コンパイラがそれを実行し、保護され、ロックが発生し、すべてがうまくいきます。ロックを取得せずに、これらの値にアクセスするコードを記述することはできません。コンパイラでは許可されません。」
これらのプロパティにより、発見されたバグの大部分が直接削除されると彼は主張しました。「これにより、これら 2 つのことが節約されます。まず、カーネル内のバグの 60% が解消されました。ありがとう。その見返りは、より早期に、より自動化された適用です。「これがレビュー時ではなくビルド時に発生した場合、私をコードを読まなければならないメンテナにして、『ああ、エラー値を適切にチェックしているね』と言わせないでください。」ああ、適切な場所にある錠前をきちんと掴んだか?』 Rust はそれを無料で提供します。これはこれまでで最高のものだ。」
たとえRustが明日消滅したとしても、カーネルはすでにCコードとインターフェースをクリーンアップすることを強制されている、とKroah-Hartman氏は主張した。彼は Rust の影響力を完全に認め、「我々はこれを Rust から盗んだ。ありがとう。これは良いアイデアなので、明日 Rust が消えても、私たちはカーネル内の C コードをかなりクリーンアップしてアイデアを取り入れました。 Linux が存在するだけで Linux がより良くなったことに感謝します。」

Rust バインディングにより、C サイドにもより深い変更が強制されています。同氏は、初期のRust-for-Linuxカンファレンスを思い出しながら、「2行のCコードを処理するために数百行のRustコード」を見たときに、「Cコードを変更できる。Cコードを変更してRustコードを単純化できる。この障壁の両側を変更することに問題はない。そうしよう。」と気づいたと語った。同氏は現在、ドライバーインターフェースを再構築したのはRust-for-Linuxチームの功績であると評価している。「関係するルール、ドライバーコアでのトレイトとインターフェースの働き、そしてカーネル内のCコードとの相互作用により、ドライバーの記述はより簡単になり、[間違った]記述はより難しくなります...彼らは素晴らしい、素晴らしい仕事をしてくれました。」
Kroah-Hartman 氏は、言語機能を超えて、Rust を信頼できないデータと「すべての入力は悪である」という考えをめぐる広範な推進に直接結びつけました。 Microsoft セキュリティからの言葉を借りて、同氏は出席者にこう語った。「私の講演で他に何も覚えていないなら、この 4 つの言葉を思い出してください。これは何年も前に Microsoft セキュリティから来たものです。彼らはすべての入力は悪であることに気づきました。すべての入力を検証する必要があります。」
彼は、「信頼できない」型ラッパーと検証 m に関する進行中の作業について説明しました。

データが信頼できないものから信頼できるものに変わる時点で明示的な検証を強制する Rust のメソッド。 「型をデータに追加するための untrusted と呼ばれるものがあり、実際にこのデータにアクセスしたいときは常に、それは単なる型であり、実際にはコードではなく、コンパイラーにヒントを与えることになります。実際にアクセスする前に、検証を通過する必要があります。」同氏によると、レビュー担当者にとっての主な利点は、移行点に「すべてが集中している」ことだという。「コードの 1 つのセクションで、ユーザー空間、信頼できないデータから信頼できるデータへの変換の検証を行うすべてのコードを 1 か所で確認できる」。

Rust will save Linux from C's inherent security weaknesses.
Linux, faced with a flood of AI-discovered security problems, could use the help.
Going forward, more and more Linux code will be written in Rust.
At the Rust Week conference , the world's biggest Rust language conference, in Utrecht, Netherlands, Linux stable kernel maintainer Greg Kroah-Hartman opened by saying: "I'm here to talk about untrusted data and Linux, and how Rust is going to save us ." After "a long month or two on the kernel security list," he pushed that point even further: "I'm going to make even a bolder statement and say, 'You are going to save Linux.' Sorry, it's all on you."
What he was talking about was the sudden flood of serious Linux security holes being discovered , such as Dirty Frag , Copy Fail , and Fragnesia , that have come to light thanks to the latest AI bug-detection programs.
As a result, Kroah-Hartman, who has "seen every single kernel security bug ever" since 2005, said the kernel team is now issuing "13 CVEs [Common Vulnerabilities and Exposures] a day, or something, something crazy." He thinks Rust is one of the few realistic ways to slash the class of bugs that come from C's traditional error-handling and resource-management pitfalls.
Also: One of the most user-friendly Linux distros I've ever used is also one of the most secure
Kroah-Hartman illustrated those pitfalls with real C bugs in the kernel, including a 15-year-old Bluetooth bug that dereferenced a pointer without checking it and a Xen bug where "we forgot to unlock" in an error path. "The majority of the bugs in the kernel are this tiny, minor stuff," he explained. "Error conditions aren't checked, locks aren't forgotten, unreleased memories leak, and vulnerabilities add up over time. They crash the kernel. This is what we live with in C. This is why we don't like it."
Kroah-Hartman argued that the "best beauty of Rust" is catching those mistakes at build time rather than in review. For example, when it comes to locking, he highlighted Rust's locking abstractions in the kernel: "The only way you can get access to inner pointers of structures is by grabbing that lock, and releasing the lock automatically. The compiler does it, it's guarded, the lock happens, everything's happy. You just can't write code to access these values...without grabbing the lock. The compiler will not let you."
Those properties, he argued, directly remove a huge fraction of the bugs he sees: "This is going to save us those two things. First, 60% of the bugs in the kernel right there, they're gone. Thank you." The payoff is earlier, more automated enforcement: "If this happens at build time, not review time, don't make me a maintainer who has to read your code [and] say, 'Oh, then you properly check that error value. Oh, did you properly grab the locks in the right spot?' Rust gives us that for free. This is the best thing ever."
Even if Rust vanished tomorrow, Kroah-Hartman argued, it has already forced the kernel to clean up C code and interfaces. He credited Rust's influence outright: "We stole this from Rust. Thank you. It's a good idea, so if Rust disappeared tomorrow, we have cleaned up the C code in the kernel so much and taken in the ideas. We thank you, you've made Linux better with it just by existing."

He described new C "guards" and scoped locks inspired by Rust: "Now we have that, we have C-scoped locks and allocators, and we've added this new code, which makes it harder for us to get it wrong, but most importantly, it makes the coding simpler." With more than 5,000 developers and only around 700 maintainers, he said, review time is the scarcest resource -- so patterns that make correctness obvious are crucial: "We optimize for making code easy to review... If I can review your code more easily, I can glance at it and say, 'Hey, look, you obviously got it right because you use that pattern. All is good, and fewer bugs.'"
Rust bindings are forcing deeper changes on the C side as well. Recalling an early Rust-for-Linux conference, he said that when he saw "hundreds of lines of Rust code to deal with two lines of C code," he realized, "I can change the C code. We can change the C code to make the Rust code simpler. We have no problem changing both sides of this barrier. Let's do that." He now credits the Rust-for-Linux team with reshaping driver interfaces: "The rules involved, and how the traits and the interfaces are working for the driver core, and the interaction with the C code in the kernel is going to make drivers simpler to write and harder to write [wrong]... They've done a great, great job."
Beyond language features, Kroah-Hartman tied Rust directly to a broader push around untrusted data and the idea that "all input is evil." Borrowing a line from Microsoft security, he told attendees: "If you never remember anything else in my talk, just remember these four words. It came from Microsoft Security many, many years ago. They realized all input is evil. You have to validate all input."
He described ongoing work on an "untrusted" type wrapper and a validate method in Rust that forces explicit validation at the point where data crosses from untrusted to trusted. "We have something called untrusted to type add it to the data, and every time you want to actually get access to this data, it's just a type, it's not actually code, it's going to hint to the compiler. You must go through validation before you actually access it." The key benefit for reviewers, he said, is that "everything's focused" on the transition points: "The one section of the code, you can see all the code that does the validation of the transformation from user space, untrusted data to trusted data in one spot."

That model extends to increasingly hostile hardware: "The model and the kernels, we don't even trust hardware. Hardware is full of bugs, but now you can have malicious hardware, and we're seeing more and more malicious hardware." His goal is to mark such data as untrusted and track it through the type system, then concentrate review energy: "This is what's going to really save us, because we don't trust data... that I think I'm going to make this unscientific quote. This is what's going to save Linux, the combination of all the other stuff and the untrusted data. I think it's going to get rid of 80% of all the CVEs we have, and that's a huge, huge number."
Now, that doesn't mean he thinks Rust is magic. It's not. He cited one of the first Rust components merged into the kernel: QR code display logic used when the kernel crashes. "That logic was written in Rust. Famously, it had a memory bug. It was given a buffer and its size, and the rest of the st code never checked the buffer size... Could scribble all over memory, because Rust can crash just as bad as C." So, Rust "is not a silver bullet."
He's also not encouraging anyone to rewrite the Linux kernel in Rust. One attendee asked, "Do you actually encourage rewriting stuff that's already there in the kernel with [Rust]?" Greg replied: "No, we don't want rewrites, so unless you're the maintainer and owner of that file, just do it for new stuff. Leave existing C code alone, and let's evolve forward after that." He gave Binder, Android's core interprocess communication (IPC) , as an example where both C and Rust implementations coexist temporarily to reach parity, after which "they're going to delete the C code, because I trust them, and they are the owners and maintainers of both those."

Over time, he expects new hardware support to skew heavily to Rust: "Rust is going to evolve in the kernel over time. Some subsystems have said, 'Hey, only new drivers for graphics should be written in Rust.'... Over time, we have now hit a critical point where you can write really good drivers for all different types of subsystems in Rust and Linux. I think over the next year or two, you will see this." In typical Kroah-Hartman fashion, he summed up the model as evolutionary: "Linux is evolution, it's not intelligent design. We evolve over time, and over time, new drivers for new hardware come in. You'll start using that new hardware. The old driver just won't get used anymore. Eventually, we'll delete them, as we do after about 20 years."
In the meantime, Kroah-Hartman told the audience: "Today, 36 million lines of Linux are in C, with 113,000 lines of Rust. Most of all, this is binding." Because drivers make up the bulk of the kernel, "Alice Ryhl, a top Rust Android programmer and the Rust-for-Linux developers, have had to go and write bindings for everything in order to write a tiny driver in Rust." That has forced the community to relax its usual rule against unused code: "We also don't like adding code to the kernel that isn't being used, so it's hard to add a binding...unless you have a user for it. So it's a bit of a chicken-and-egg problem. We've had to bend that rule a little bit."
He also flagged real-world deployments coming soon: "Android phones will be coming out, hopefully soon, running all the [Rust] code in Binder. So that's again, billions of devices can be running Rust in Linux soon."
What ultimately sold a number of core maintainers, including him, on Rust was how it "makes reviewing code easier." With CI [Continuous Integration] bots enforcing builds and Rust's type system enforcing key invariants, maintainers can "focus on the logic" rather than resource bookkeeping: "I can care about that one function. I don't have to worry about the rest of this stuff, because I assume that it works properly, because it was built properly."
