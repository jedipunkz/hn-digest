3 つの層、1 つの不変要素 — プラグインの不透明度:
コア — タスク エンジン、オーケストレーター、セーフティ レイヤー、イベント バス、デーモン。指揮者。
アダプター — コントラクト ( TriggerAdapter 、 CommunicationAdapter 、 AgentAdapter 、 GitHostingAdapter )。スコア。
プラグイン — 各アダプターの背後にある交換可能な実装。楽器たち。
コアは各アダプターにプラグインが登録されていることを確認します。それ以上のことは行いません。具体的な実装は不透明です。アダプターのコントラクトが統合境界になります。 GitHub を GitLab に、Telegram を Slack に、Claude コードを OpenCode に交換します。コアのコードは変更されません。プラグインの不透明度は意図的に行われます。エンジニアは、ユーザーが持参したあらゆるツールに接続します。アーキテクチャ テストは、境界を越えた瞬間にビルドを中断します。
タスクがデーモンを介してどのように移動するか:
トリガーとスケジュール。デーモンはトリガー プラグインをポーリングします。新しい作業は、OS スケジューラーに触発されたステート マシン上のタスクとして実行されます。優先順位によって順序が設定され、より緊急なものが到着するとプリエンプションが行われます (プリエンプションされたタスクはフェーズ間で一時停止し、最後のチェックポイントから再開されます)。クラッシュと一時的な障害は、制限されたバックオフで再試行されます。
ワークスペースを隔離します。各タスクは独自の git worktree を取得します。同時実行は衝突せず、メインのチェックアウトはそのまま残ります。
エンジニアリング パイプラインを実行します。 6 つのフェーズ — 要件 → 調査 → 計画 → 実行 → レビュー → 納品 — コンテキストを伝えるチェックポイント付きセッション ジャーナルを備えています。パイプラインのどこでも、エージェントが単独では解決できない問題に遭遇した場合 - mos

通常、要件の間、タスクはブロック状態に移行します。システムは、タスクのブロックを解除できる人に通信プラグインを介して連絡し、その人が応答すると作業が再開されます。
準備ができるまで繰り返します。レビューで根本原因が上流にあることが判明した場合、作業は以前のフェーズに戻される可能性があります。エージェントが結果を報告し、オーケストレーターがルートを決定します (フェーズ自体を選択することはありません)。プル リクエストが開いたら、コメント、CI エラー、マージ競合を確認し、型指定されたイベントとしてパイプラインに再入力します。新しいプッシュが行われるたびに、古い承認が却下されます。やり直しには限界があります。上限を超えると、タスクはエスカレートしていきます。
サインオフ後にマージします。人間は、要件、重要な決定、最終的な承認について常に最新情報を把握しています。プル リクエストが CI グリーンで承認されると、エンジニアはそれを自動的にマージできます。自動マージはデフォルトでオフになっているため、デフォルトでは承認によってタスクが完了し、マージが行われます。また、外部マージも検出されて完了します。

エンジニアは、分離された git ワークツリー内で CLI エージェントをサブプロセスとして実行します。 Core は、CLI 内の個々のファイルの書き込みや git コマンドをインターセプトすることはできませんが、エージェントがいつ実行されるか、およびそのコストは制御されます。
自律的な操作では、CLI が独自の権限プロンプトをスキップする必要があります。たとえば、Claude Code は --dangerously-skip-permissions で起動します。ワークツリーは包含境界であり、CLI プロンプトではありません。
ステートマシン。すべてのエージェントの呼び出しはタスク エンジンを通過し、タスク エンジンはタスクの現在の状態に基づいて許可または拒否します。 requirements_gathering 内のタスクは実行をトリガーできません。失敗したタスクは、再試行を設計するまで実行できません。
コストの上限。

すべてのエージェント呼び出しで、cost.incurred イベントが生成されます。支出は、タスクごと、日ごと、月ごとに加えて、プロバイダーごとのリクエスト予算に加算されます。制限の 80% になると、エンジニアは警告します。 100% に設定すると、次のエージェントの通話でさらに費用が発生する前に、実行中の作業が終了し、すぐに通知されます。タスクごとまたはプロバイダーごとに違反が発生すると、そのタスクが終了し、それについて DM が送信されます。毎日または毎月の世界規模の違反が発生すると、実行中のすべてのタスクが一度に終了し、制限と停止したタスクの数を示す単一のアラートが送信されます。タスクごとに 1 つの DM が送信されることはありません。
自律性のエスカレーション。エージェントは、自身が行った任意の決定 (名前の変更、新しい依存関係、構造の変更) を表面化し、Core は決定ごとに safety.yaml の自律性ポリシーを参照します。エージェントは、許可されたところからは黙って続行し、カテゴリごと、サイズのしきい値とともに、指示された場所を尋ねるために一時停止します。 (自律性構成を参照してください。)
スコープ境界 (CLI がアクセスできるファイルとブランチ) は safety.yaml で構成できますが、実行時にはまだ適用されません。現在、コアは CLI の内部ファイル書き込みをゲートできません。 (マージ ポリシー、コスト上限、自律性エスカレーションが適用されます。) ランタイム スコープの適用とサンドボックス タスクの実行がロードマップにあります。次に計画されている内容については、docs/future-considerations.md を参照してください。

An autonomous orchestrator for AI coding agents. The Engineer drives Claude Code, OpenCode, and other CLI agents through the full engineering lifecycle — from task intake to merged pull request — with safety rails, a full audit trail, and a swappable plugin architecture for any tool you use.
One issue, one label — then it grounds itself, asks when it's unsure, ships a pull request, and merges on your approval. Intake to delivery.
v1.0.0 — it works, end to end.
The Engineer runs the full pipeline today: task intake → requirements → research → planning → execution → review → delivery. It's young and built by one person, so expect rough edges — use it, read the code, file issues , and help shape where it goes. The build journal traces how it got here.
AI coding agents: if you're an AI agent working on this codebase — not just reading about it — AGENTS.md is your required entry point. Read and follow it before making any change.
AI coding CLIs like Claude Code, OpenCode, and other AI coding agents are extraordinarily capable inside a single prompt. They fall short the moment you want them to do real engineering work — receiving tasks, gathering requirements, researching across the codebase, the web, and adjacent systems, planning the approach, executing safely, self-reviewing, shipping a pull request that survives review, iterating on review comments and CI failures, and merging only after sign-off.
The Engineer is the orchestration layer that closes that gap. It runs as a long-lived daemon that listens for tasks wherever engineering work actually lives — GitHub Issues today; Jira, Azure DevOps, and other popular ticket-management systems as plugin support lands.
From there it drives a coding CLI through that lifecycle end-to-end — preserving context across phases, isolating each task in its own workspace, and reaching out through your communication plugins when it hits a blocker or needs a decision.
Humans stay in the loop where it matters — requirements, key decisions, and the final review — by design, not by omission. Every action is observable, every plugin is swappable, every decision is auditable.
Think of it as the conductor . The coding CLIs are the instrumentalists.
Prerequisites: Node.js 22+, pnpm
git clone https://github.com/FarzamMohammadi/the-engineer.git && cd the-engineer
pnpm run setup # install dependencies, build, link the `engineer` CLI
engineer start # first-run setup, then start the daemon
pnpm run setup confirms before it acts and is safe to re-run any time. If pnpm's global bin directory isn't configured yet, it offers to set that up and tells you what to do next. engineer start walks you through first-run configuration, then runs the daemon in the foreground — Ctrl+C to stop. When engineer start reports the daemon is ready, the dashboard is at http://localhost:3847 . If anything misbehaves, docs/troubleshooting.md walks symptom → fix.
Dev mode (without a global install): use pnpm dev <command> in place of engineer .
AI agents (no TTY): engineer start prompts interactively and can't be driven headless. Configure non-interactively with engineer start --seed <dir> — copy seed-example/ , fill in values, pass the directory. Start at AGENTS.md , your entry point.
engineer start # Setup (first run) + start daemon
engineer stop # Graceful shutdown
engineer status # Daemon state + task listing (IDs, state, title, age)
engineer status --all # Include completed and failed tasks
engineer logs # View daemon logs (--follow, --raw)
engineer doctor # Health checks
engineer why < task-id > # Explain a task's decision trail
engineer retry < task-id > # Re-queue a blocked or failed task
engineer cancel < task-id > # Cancel a task that hasn't finished
<task-id> accepts a full ULID or a unique prefix — copy the short form from engineer status .
All commands accept --home <path> to use a custom data directory instead of ~/.engineer .

Three tiers, one invariant — Plugin Opacity:
Core — task engine, orchestrator, safety layer, event bus, daemon. The conductor.
Adapters — contracts ( TriggerAdapter , CommunicationAdapter , AgentAdapter , GitHostingAdapter ). The score.
Plugins — swappable implementations behind each adapter. The instruments.
Core verifies each adapter has a plugin registered — nothing more. The specific implementation is opaque; the adapter contract is the integration boundary. Swap GitHub for GitLab, Telegram for Slack, Claude Code for OpenCode — Core's code does not change. Plugin Opacity is deliberate: The Engineer plugs into any tooling you bring, and an architecture test breaks the build the moment that boundary is crossed.
How a task moves through the daemon:
Trigger and schedule. The daemon polls the trigger plugin. New work lands as a task on an OS-scheduler-inspired state machine — priority sets the order, with preemption when something more urgent arrives (preempted tasks pause between phases and resume from their last checkpoint). Crashes and transient failures retry with bounded backoff.
Isolate the workspace. Each task gets its own git worktree — concurrent runs don't collide, and your main checkout stays untouched.
Run the engineering pipeline. Six phases — requirements → research → planning → execution → review → delivery — with a checkpointed session journal carrying context across them. Anywhere in the pipeline, if the agent hits something it can't resolve alone — most commonly during requirements — the task moves to blocked . The system reaches out via your comms plugins to whoever can unblock the task, and work resumes when they respond.
Iterate until ready. Work can be handed back to an earlier phase when review finds the root cause lives upstream — the agent reports an outcome and the orchestrator decides the route (it never picks a phase itself). After the pull request opens, review comments, CI failures, and merge conflicts re-enter the pipeline as typed events. Each new push dismisses stale approvals. Rework is bounded — past the ceiling, the task escalates to you.
Merge after sign-off. Humans stay in the loop on requirements, key decisions, and the final approval. When a pull request is approved with CI green, The Engineer can merge it automatically — auto-merge is off by default, so by default the approval completes the task and you merge — and an external merge is detected and finalized too.

The Engineer runs CLI agents as subprocesses inside isolated git worktrees. Core can't intercept individual file writes or git commands inside the CLI — but it controls when the agent runs and what it costs .
Autonomous operation requires the CLI to skip its own permission prompts — Claude Code launches with --dangerously-skip-permissions , for example. The worktree is the containment boundary, not in-CLI prompts.
State machine. Every agent invocation passes through the Task Engine, which permits or denies it based on the task's current state. A task in requirements_gathering can't trigger execution; a failed task can't run until you engineer retry .
Cost ceilings. Every agent call emits a cost.incurred event. Spending accumulates per task, per day, per month, plus per-provider request budgets. At 80% of any limit, The Engineer warns. At 100%, it terminates the in-flight work and notifies you immediately — before the next agent call can accrue more spend. A per-task or per-provider breach terminates that one task and DMs you about it; a global daily or monthly breach terminates every in-flight task at once and sends a single alert naming the limit and how many tasks it stopped — never one DM per task.
Autonomy escalation. The agent surfaces discretionary decisions it makes (a rename, a new dependency, a structural change), and Core consults your safety.yaml autonomy policy per decision: it proceeds silently where you let it, and pauses to ask you where you said to — by category, with size thresholds. (See autonomy configuration .)
Scope boundaries (which files and branches the CLI may touch) are configurable in safety.yaml but not yet enforced at runtime — Core can't gate the CLI's internal file writes today. (Merge policy, cost ceilings, and autonomy escalation are enforced.) Runtime scope enforcement and sandboxed task execution are on the roadmap — see docs/future-considerations.md for what's planned next.
