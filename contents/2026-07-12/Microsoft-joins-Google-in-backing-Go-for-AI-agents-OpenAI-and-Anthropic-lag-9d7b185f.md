Go はクラウド インフラストラクチャの共通語として登場し、コンテナ オーケストレーションや CI/CD パイプラインからエンジニアが日常的に使用するコマンドライン ツールに至るまで、あらゆるものに使用されています。 Kubernetes、Docker、Terraform はすべてその中に書かれており、バックエンド サービスを構築するチームにとってはデフォルトの選択肢として残っています。
同時に、AI エージェントはほぼすべての主要なソフトウェア ベンダーの焦点となっており、各ベンダーは開発者に AI エージェントを構築するためのネイティブな方法を提供しようと競い合っています。だからこそ、Microsoft は現在、エージェント フレームワークを Go に導入し、クラウド ネイティブ開発者が他のあらゆる用途ですでに使用している言語で AI エージェントを構築できるファーストパーティの方法を提供しています。
金曜日からパブリック プレビューで利用可能になる Microsoft Agent Framework for Go は、基本的に、Python や .NET の対応物で既に利用可能なものと同じ構成要素の多くを Go 開発者に提供します。これには、Microsoft Foundry、Azure OpenAI、Anthropic、Gemini のモデルのサポート、エージェントを外部システムに接続するためのツール呼び出しと MCP のサポート、および複数の連携機能が含まれます。

タスクに協力するエージェント。
「Microsoft Agent Framework は、単一プロンプト コールから運用エージェント システムに移行しようとしている開発者向けに設計されています。」
MicrosoftのシニアソフトウェアエンジニアであるQuim Muntal氏は、金曜日のリリースを発表したブログ投稿の中で、Agent Framework自体は、より高性能で長時間実行されるAIシステムを構築する人向けであると述べた。
「Microsoft Agent Framework は、単一のプロンプト コールから運用エージェント システムに移行しようとしている開発者向けに設計されています」と Muntal 氏は書いています。 「ツールを使用し、コンテキストを保持し、他のエージェントと調整し、結果をストリーミングするエージェントは、実際のアプリケーションの一部として監視および管理できます。」
それは、サービス、コマンドライン ツール、バックグラウンド ワーカー、および言語が長い間デフォルトとなっているクラウド ネイティブ アプリケーションなど、Go がすでに占めている領域と同じです。 Go SDK は、その領域で構築する開発者の手にこれらの機能を直接提供します。
Microsoft のエージェント フレームワークが Go に導入された経緯
簡単にまとめると、Microsoft は 2025 年 10 月に、AI エージェントとマルチエージェント システムを構築するためのオープンソース ツールキットとして Agent Framework を導入し、以前の 2 つの Microsoft プロジェクトである AutoGen と Semantic Kernel を単一のサポート対象プラットフォームに統合しました。
今年の 4 月に一般公開され、6 月の Microsoft の Build カンファレンスで、同社は多くの新機能とツールを追加しました。コンテキスト管理や人間による承認ステップなどの運用パターン用の「エージェント ハーネス」、Microsoft Foundry を介したホスト エージェント、CodeAct と呼ばれるより高速なツール呼び出し方法、および新しいハンドオフ パターンを介した複数のエージェントの連鎖のサポートです。
そのすべてを通じて、エージェント フレームワークは .NET と Python でのみ利用可能なままでした。コミュニティの中には Go のサポートをリクエストする人もいました。

Rust については、2025 年 10 月に Microsoft のプロダクト マネージャーがフレームワークの Go バージョンを検討していることを認めましたが、それは「少なくとも数か月先」でした。
「これは実稼働グレードのオーケストレーションであり、チャット ループ ラッパーではありません。」
まだ初期の段階ですが、開発者コミュニティからは、このリリースが何を意味するのか、何を意味しないのかについて、すでにいくつかのつぶやきが上がっています。 LinkedIn の投稿 の中で、AI エンジニアの Pratik Dhanave 氏は、これを Go-first エンジニアにとってのマイルストーンと呼び、条件付きルーティング、サブワークフロー、チェックポイント、人間参加型レビューなどのパターンをサポートする SDK のグラフベースのワークフロー オーケストレーションを取り上げています。
「これは本番グレードのオーケストレーションであり、チャット ループ ラッパーではありません」と彼は書いています。
Dhanave 氏はまた、Go SDK がまだ実行していないことについての Microsoft 独自のドキュメントについても指摘しています。ハンドオフ オーケストレーションや CodeAct などの機能は、.NET で引き続き利用できますが、まだ Go には移植されていません。 「パブリック プレビューでのこのレベルの率直さには感謝しています」と彼は続けます。
Go における Google、OpenAI、Anthropic の立場
Google も、ペースは異なるものの、同様の軌跡をたどっています。同社は 2025 年 4 月に Python 専用ツールキットとして独自のエージェント開発キット (ADK) をデビューさせ、その後同年 11 月に Go サポートを追加し、続いて今年 3 月に完全な正式な 1.0 をリリースしました。
もちろん、Go は Google の製品であり、2009 年に同社のオープンソース保管庫から出現しました。この動きは、その後のクラウド ネイティブの世界の多くを形成しました。その後、Go は最新のインフラストラクチャの構築と展開方法を定義した 2 つのプロジェクトである Kubernetes と Docker の背後にある言語となり、プラットフォーム エンジニアにとって Go がデフォルトの選択肢となるのに役立ちました。その引き寄せは、Google をはるかに超えて、AWS、Cloudflare、Microsoft などの領域にまで広がりました。

テクノロジー界の他の無数の巨人。
この魅力を考えると、エージェント ツールにおけるネイティブ Go サポートの需要は驚くべきことではありません。エージェント サービスを構築したり、既存のマイクロサービスに AI 機能を埋め込んだり、独自の意思決定を行うインフラストラクチャ ツールを作成したりしたい Go 開発者は、独自の言語で公式の SDK を使用せずに対処する必要がありました。つまり、生の HTTP 呼び出しに手を伸ばしたり、Go サービスと一緒に別の Python または Node.js プロセスを起動したり、品質やサポートが均一でない非公式のコミュニティ管理ライブラリに依存したりする必要がありました。
したがって、Microsoft が確立された開発者ベースに対応していることは明らかです。しかし、AI 分野の他の大物企業はまだ追随していません。今日の時点では、コミュニティからのサポート要求にもかかわらず、Anthropic の Claude Agent SDK も OpenAI の Agent SDK も Go を正式にサポートしていません。
Microsoft は独自に AI エージェントを Go に導入しているわけではありません。しかし、その登場は、2 つの最大手のクラウド ベンダーが、業界で最も広く使用されているインフラストラクチャ言語の 1 つに対するファーストパーティ エージェント フレームワークを提供することを意味します。基盤モデル分野に残された 2 つの最大手である Anthropic と OpenAI は遅れをとっています。

Go has emerged as the lingua franca for cloud infrastructure , used for everything from container orchestration and CI/CD pipelines to the command-line tools engineers rely on daily. Kubernetes, Docker, and Terraform are all written in it, and it remains a default choice for teams building backend services.
At the same time, AI agents have become the focus of nearly every major software vendor, each racing to give developers a native way to build them. And that’s why Microsoft is now bringing its Agent Framework to Go, giving cloud-native developers a first-party way to build AI agents in the language they already use for everything else.
Available in public preview from Friday, Microsoft Agent Framework for Go essentially gives Go developers many of the same building blocks already available to their Python and .NET counterparts: support for models from Microsoft Foundry , Azure OpenAI, Anthropic, and Gemini, tool-calling and MCP support for connecting agents to external systems, and the ability to coordinate multiple agents working together on a task.
“Microsoft Agent Framework is designed for developers who are moving from single prompt calls to production agent systems.”
In a blog post announcing the release on Friday, Quim Muntal , a senior software engineer at Microsoft, notes that the Agent Framework itself is for those building more capable, longer-running AI systems.
“Microsoft Agent Framework is designed for developers who are moving from single prompt calls to production agent systems,” Muntal writes. “Agents that use tools, keep context, coordinate with other agents, stream results, and can be observed and governed as part of real applications.”
That’s the same ground Go already occupies: services, command-line tools, background workers, and cloud-native applications where the language has long been the default. The Go SDK puts those capabilities directly in the hands of developers building in that space.
How Microsoft’s agent framework arrived in Go
By way of a brief recap, Microsoft introduced Agent Framework in October 2025 as an open-source toolkit for building AI agents and multi-agent systems, unifying two earlier Microsoft projects, AutoGen and Semantic Kernel , into a single supported platform.
It reached general availability this past April , and at Microsoft’s Build conference in June, the company added a bunch of new features and tooling: an “agent harness” for production patterns like context management and human approval steps, hosted agents through Microsoft Foundry, a faster tool-calling method called CodeAct, and support for chaining multiple agents together through a new handoff pattern.
Throughout all of that, Agent Framework remained available only in .NET and Python. Some in the community had requested support for Go , as well as Rust, with a Microsoft product manager confirming way back in October 2025 that it was considering a Go version of the framework, but it was “at least a few months off.”
“This is production-grade orchestration, not a chat loop wrapper.”
It’s still early days, but there are already some murmurings from the developer community about what the release does, and doesn’t, mean. In a LinkedIn post , AI engineer Pratik Dhanave calls it a milestone for Go-first engineers, singling out the SDK’s graph-based workflow orchestration, which supports patterns like conditional routing, subworkflows, checkpointing, and human-in-the-loop review.
“This is production-grade orchestration, not a chat loop wrapper,” he writes.
Dhanave also points to Microsoft’s own documentation of what the Go SDK doesn’t yet do: handoff orchestration and CodeAct, among other features, remain available in .NET but not yet ported to Go. “I appreciate this level of candor in a public preview,” he continues.
Where Google, OpenAI, and Anthropic stand on Go
Google, for its part, has been following a similar trajectory, albeit at a different cadence. The company debuted its own Agent Development Kit (ADK) in April 2025 as a Python-only toolkit, then added Go support that November , followed by its full formal 1.0 launch this past March .
Go, of course, is a product of Google, emerging from the company’s open-source vaults back in 2009 , a move that shaped much of the cloud-native world that followed. It went on to become the language behind Kubernetes and Docker, two projects that defined how modern infrastructure is built and deployed, which helped make Go the default choice for platform engineers. That pull extended well beyond Google into the realm of AWS , Cloudflare, Microsoft, and countless other giants of the tech world.
Given that pull, demand for native Go support in agent tooling is hardly surprising. Go developers wanting to build agent services, embed AI capabilities into existing microservices, or write infrastructure tooling that can make its own decisions have had to make do without an official SDK in their own language: reaching for raw HTTP calls, spinning up a separate Python or Node.js process alongside their Go service, or relying on unofficial, community-maintained libraries of uneven quality and support.
So Microsoft is clearly catering to a well-established developer base. Other big-hitters in the AI realm, however, aren’t yet following suit. As of today, neither Anthropic’s Claude Agent SDK nor OpenAI’s Agents SDK officially support Go, despite community requests for such support .
Microsoft isn’t bringing AI agents to Go on its own. But its arrival means two of the biggest cloud vendors now offer first-party agent frameworks for one of the industry’s most widely used infrastructure languages. Anthropic and OpenAI, the two biggest names left in the foundation model space, are lagging.
